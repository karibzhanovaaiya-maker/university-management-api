Introduction

One Spring Boot backend + one static Next.js frontend, behind nginx, talking to PostgreSQL.

The system is a classic layered (n-tier) web application:

  • Presentation — a static Next.js UI (browser) and a REST API (Spring MVC controllers).
  • Business — Spring @Service classes hold the rules (validation, the "can't delete a teacher with courses" rule, etc.).
  • Persistence — Spring Data JPA repositories map entities to PostgreSQL tables.
  • Cross-cutting — Spring Security (JWT) for auth, a global exception handler for errors, OpenAPI for docs.
Diagrams are rendered live with Mermaid. Scroll horizontally on small screens. Every diagram below is followed by a written explanation.

Explanation

  • Guest (no token) can only register/login and read the public Swagger docs.

  • Authorisation is enforced in SecurityConfig: GET /api/courses is for any authenticated user; all writes and the other resources are ADMIN-only.

2 · Deployment diagram

What runs where, and how a request travels.

flowchart TD
  B["💻 Browser"] -->|HTTPS| CF["☁️ Cloudflare (edge TLS)"]
  CF -->|HTTPS| NG["nginx :443 (reverse proxy)"]
  NG -->|"/ (static files)"| FE["Next.js export — out/"]
  NG -->|"/api · /auth · /swagger-ui"| API["Spring Boot app :8080 (java -jar, background service)"]
  API -->|"JDBC localhost only"| DB[("PostgreSQL :5432")]
  classDef n fill:#161b22,stroke:#30363d,color:#e6edf3;
  class B,CF,NG,FE,API,DB n;

Explanation

  • The browser only ever talks to one origin (karibzhanova.alma-ai.cc) → no CORS.
  • Cloudflare terminates TLS at the edge; nginx on the server splits traffic by path.
  • / → static Next.js files (no Node process). /api, /auth, /swagger-ui → the Java app.
  • The Java app is a single executable JAR run continuously as a background service (auto-start on boot, auto-restart on crash), listening on internal port 8080.
  • PostgreSQL is bound to localhost only — never exposed to the internet.

3 · Layered architecture (component diagram)

How a call flows through the backend layers.

flowchart TD
  REQ["HTTP request"] --> SEC["Security filter chain
JwtFilter + SecurityConfig"]
  SEC --> CTRL["Controllers
@RestController"]
  CTRL -->|uses| DTO["DTOs
(request / response)"]
  CTRL --> SVC["Services
@Service @Transactional"]
  SVC --> REPO["Repositories
JpaRepository"]
  REPO --> ENT["Entities
@Entity"]
  ENT --> DB[("PostgreSQL")]
  SVC -.->|throws| EXC["Exceptions"]
  EXC --> GEH["GlobalExceptionHandler
@RestControllerAdvice"]
  GEH -->|JSON error| OUT["HTTP response"]
  CTRL -->|JSON| OUT
  classDef n fill:#161b22,stroke:#30363d,color:#e6edf3;
  class REQ,SEC,CTRL,DTO,SVC,REPO,ENT,DB,EXC,GEH,OUT n;

Explanation

  • Every request first passes the security filter chain (the JWT is validated here).
  • Controllers accept/return DTOs (never raw entities) and delegate to services.
  • Services hold business logic and run inside a transaction; they call repositories.
  • Repositories (Spring Data JPA) translate method calls into SQL against entities.
  • Any exception is caught centrally by the GlobalExceptionHandler and turned into a uniform JSON error with the right HTTP status.

  }
  Student "1" --> "1" User : OneToOne
  Teacher "1" --> "*" Course : OneToMany
  Course "*" --> "1" Teacher : ManyToOne
  Student "*" --> "*" Course : ManyToMany
  UserFile "*" --> "1" User : ManyToOne

Explanation

  • User ↔ Student — one-to-one. The FK user_id is unique so no two students share a user.
  • Teacher ↔ Course — one-to-many. A course belongs to one teacher; a teacher has many courses.
  • Student ↔ Course — many-to-many. Student is the owning side (declares the @JoinTable student_courses); Course is the inverse (mappedBy).
  • UserFile → User — many-to-one (avatars; Lab 8).
  • All associations are LAZY — related rows load only when actually used (Lab 3).

Explanation

7 · Sequence — Login (JWT issuance, Lab 7)

sequenceDiagram
  actor U as Browser
  participant AC as AuthController
  participant AM as AuthenticationManager
  participant UDS as CustomUserDetailsService
  participant DB as PostgreSQL
  participant J as JwtUtil
  U->>AC: POST /auth/login {userName, password}
  AC->>AM: authenticate(userName, password)
  AM->>UDS: loadUserByUsername(userName)
  UDS->>DB: findByUserName(userName)
  DB-->>UDS: User (BCrypt hash, role)
  UDS-->>AM: UserDetails
  AM-->>AC: Authentication OK (password matches)
  AC->>J: generateToken(userName)
  J-->>AC: signed JWT (HS, subject/iat/exp)
  AC-->>U: { token, userName, role }

Explanation

  • The AuthenticationManager loads the user and checks the BCrypt password.
  • On success, JwtUtil.generateToken() returns a signed token (claims: subject, issuedAt, expiration).
  • The browser stores the token and sends it as Authorization: Bearer … on later requests.
  • Bad credentials → AuthenticationException → 401 (handled globally).

8 · Sequence — Secured request (GET /api/courses)

sequenceDiagram
  actor U as Browser
  participant NG as nginx
  participant F as JwtFilter
  participant SC as SecurityContext
  participant CO as CoursesRestController
  participant SV as CourseService
  participant RE as CoursesRepository
  participant DB as PostgreSQL
  U->>NG: GET /api/courses (Bearer token)
  NG->>F: proxy to :8080
  F->>F: validate JWT (JwtUtil)
  alt missing/invalid token
    F-->>U: 401 Unauthorized
  else valid
    F->>SC: set Authentication (role)
    F->>CO: continue filter chain
    CO->>SV: findAll()
    SV->>RE: findAll()
    RE->>DB: SELECT * FROM courses
    DB-->>RE: rows
    RE-->>SV: List~Course~
    SV-->>CO: entities (lazy init inside tx)
    CO-->>U: 200 List~CourseResponse~
  end

Explanation

  • JwtFilter runs once per request, validates the token and puts the authenticated principal (with role) into the SecurityContext.
  • No/invalid token on a protected path → 401. Wrong role → 403.
  • The service initialises any needed lazy associations inside the transaction, then the controller maps entities to response DTOs.

9 · Sequence — Create student (validation + relation, Labs 4/5)

sequenceDiagram
  actor A as Admin
  participant SC as StudentsRestController
  participant V as Bean Validation
  participant SV as StudentService
  participant UR as UsersRepository
  participant SR as StudentsRepository
  participant GEH as GlobalExceptionHandler
  A->>SC: POST /api/students (@Valid StudentFormDto)
  SC->>V: validate DTO
  alt invalid input
    V-->>GEH: MethodArgumentNotValidException
    GEH-->>A: 400 Bad Request + field errors
  else valid
    SC->>SV: create(form)
    SV->>UR: findById(userId)
    alt user not found
      UR-->>SV: Optional.empty()
      SV-->>GEH: EntityNotFoundException
      GEH-->>A: 404 Not Found
    else user exists
      SV->>SR: existsByUserId / save(student)
      SR-->>SV: Student
      SV-->>SC: Student
      SC-->>A: 201 Created (StudentResponse)
    end
  end

Explanation

  • Bean validation (@NotBlank, @Email, @Min…) runs first; failures become a 400 with per-field messages.
  • The service links the student to an existing user; a missing user id → 404. (The UI prevents this by offering a user dropdown.)
  • A user already linked to a student → 409 via BusinessException.

10 · Sequence — Delete teacher (business rule, Labs 5/9)

sequenceDiagram
  actor A as Admin
  participant TC as TeachersRestController
  participant TS as TeacherService
  participant TR as TeachersRepository
  participant GEH as GlobalExceptionHandler
  A->>TC: DELETE /api/teachers/{id}
  TC->>TS: delete(id)
  TS->>TR: findById(id)
  TR-->>TS: Teacher (+ courses)
  alt teacher has courses
    TS-->>GEH: BusinessException
    GEH-->>A: 409 Conflict
  else no courses
    TS->>TR: delete(teacher)
    TC-->>A: 204 No Content
  end

Explanation

  • The rule "a teacher with assigned courses cannot be deleted" lives in TeacherService.delete().
  • Violation → BusinessException → 409 Conflict. This exact rule is covered by a unit test (Lab 9).

11 · Activity diagram — request lifecycle

The decision flow every API call goes through.

flowchart TD
  S([Incoming request]) --> TLS[Cloudflare + nginx HTTPS]
  TLS --> R{Path?}
  R -->|"static page"| ST[Serve Next.js file] --> E([Response])
  R -->|"api or auth"| PUB{Public endpoint?}
  PUB -->|"yes"| CTRL
  PUB -->|"no"| TOK{Valid JWT?}
  TOK -->|"no"| E401[401 Unauthorized] --> E
  TOK -->|"yes"| ROLE{Role allowed?}
  ROLE -->|"no"| E403[403 Forbidden] --> E
  ROLE -->|"yes"| CTRL[Controller to Service to Repository]
  CTRL --> VAL{Valid and rules OK?}
  VAL -->|"no"| ERR[GlobalExceptionHandler 400 404 409] --> E
  VAL -->|"yes"| DBOP[(DB read write)] --> OK[200 201 204 JSON] --> E

Explanation

Static pages are served straight from disk. API calls are gated by authentication then authorisation, then run through the controller→service→repository chain. Validation and business-rule failures are converted to the correct HTTP status by the global handler; success returns the appropriate 2xx with a JSON body.
